# Консольное приложение для параллельной загрузки файлов по HTTP/HTTPS 

## Основная задача:
### Написание многопоточного клиента HTTP/HTTPS без библиотеки curl и её производных  

## Библиотеки
### OpenSSL https://www.openssl.org/  

## Требования  
### Общие  
- **CMake** 3.10 или выше  
- **C++ компилятор** с поддержкой C++17  

## Локальная установка и запуск основного консольного приложения
### Клонирование репозитория  
git clone https://github.com/user-sw-67/bolid_task.git  
cd bolid_task/solution  
### Создание папки для сборки и переход в неё  
mkdir build  
cd build/  
### Компиляция  
cmake ..  
make  
### Запуск  
./loader <файл с url> <директория> <кол-во одновременных загрузок>

## Автоматизация тестирования
### Для проведения замеров производительности создан скрипт  
#### Задачи скрипта    
- Генерирует __links__.txt со случайными URL из variants.txt  
- Собирает основной проект loader  
- Запускает его N раз с заданными параметрами  
- Сохраняет логи в log/1.log, log/2.log, ...  
- Выводит время выполнения каждого запуска

## Небольшой анализ
### Показатели запусков  
- Всего запусков программы: 111  
- Всего успешных запусков: 80   
- Аварийных завершений: 31  

### Итоговые выводы  
1. **Многопоточность эффективна**: увеличение потоков с 1 до 100 дает ускорение до 49 раз, с 1 до 1000 до 54 раз   
2. **Эффект насыщения**: 1000 потоков почти не дают прироста по сравнению со 100 (максимум +10%)  
3. **Стабильность растет с потоками**: разброс времени при 100+ потоках в 10-40 раз меньше, чем при 1 потоке  
4. **Аварийность**: 31 завершение из 111 запусков (28%) связано с ошибками сети/DNS/SSL, что характерно для больших объемов данных  
5. **Оптимальность**: 100 потоков — лучшее соотношение скорости и стабильности  

### Почему происходили аварийные завершения?
- По условию задачи, программа должна завершать работу при:
    - Ошибках DNS (не могу найти хост)
    - Ошибках SSL рукопожатия
    - Проблемах с сетевым соединением

### Вывод по разбросу:
- При 1 потоке разброс огромен (до 42 секунд при 250 ссылках)
    - Один зависший запрос сильно влияет на общее время
- При 10 потоках разброс значительно меньше (макс 4.7 сек)
    - Параллелизм сглаживает отдельные задержки
- При 100+ потоках разброс минимальный (менее 2.5 сек)
    - Высокая стабильность результатов

### Вывод по ускорению:
- Переход с 1 на 10 потоков дает ускорение в 5-9 раз
- Переход с 10 на 100 потоков дает ускорение в 1.3-5.5 раз (эффект снижается)
- Переход с 100 на 1000 потоков дает минимальный прирост (0-10%)
